# 路线图

这份路线图按当前共识写，不按旧愿景写。

## 阶段 0：文档重置

目标：

- 停止沿用旧叙事继续扩张
- 把当前共识写清楚
- 给后面开发一个统一判断基线

完成标准：

- 新文档入口可读
- 历史文档已归档
- 当前方向、包策略、开发策略和路线图都写清

## 阶段 1：替换当前脚本体系

目标：

- 工具链方向先收敛
- 明确用 `tsx + esbuild` 完全取代当前 `scripts/` 聚合脚本和 Python 脚本
- 先做成一套独立的脚本体系
- 不再继续把仓库工作流塞进 `automation` / `repo-command`

默认分工：

- `tsx`：仓库内部任务入口、参数解析、任务编排
- `esbuild`：构建

完成标准：

- 当前 `scripts/workflow.py` 这类聚合入口已经被新脚本体系替代
- 新入口不依赖 `automation` / `repo-command`
- 新脚本体系不直接依赖当前项目包实现
- 构建和日常开发路径的职责拆清
- 仓库中不再保留 Python 脚本作为主工作流入口

本阶段的具体收敛方向：

- 用 `tsx` 承接仓库内部任务入口、参数解析和任务编排
- 用 `esbuild` 承接构建
- 让清理、生成和维护任务也统一迁到 TypeScript 工具链
- 先做成独立的脚本体系，不直接依赖当前项目包
- 脚本体系的目标是可脱离当前仓库结构单独存在，不把 `protocol`、`std`、`cli` 等包当内部实现前提
- 为了保持独立性，允许在脚本体系里重写一份相同思想的类型和方法
- 允许重复，不强求复用；独立性优先于复用
- 把独立脚本体系同时作为新设想的试点位
- 新设想先在脚本体系里验证，再决定是否进入主线包或代码规范
- 不再继续扩 `scripts/` 里的大聚合入口
- 完全消除 Python 脚本
- 不把仓库私有工作流继续推向 `automation` / `repo-command`

这阶段对“独立”的定义是：

- 不依赖当前仓库包的内部实现
- 不把当前 monorepo 的路径结构写死成脚本前提
- 不因为想复用现有代码，把脚本体系反向绑死在当前工程上
- 只继承代码规范和设计风格，不继承实现耦合
- 可以复制当前协议和工具写法，但复制后的实现由脚本体系自己维护

这阶段对“试点”的定义是：

- 可以在独立脚本体系里试后续的新设想
- 可以先验证新的类型结构、工具写法、任务组织方式
- 试点通过后，再决定是否上升到 `protocol`、`std` 或代码规范
- 不要反过来为了试点先改主线包

## 阶段 2：协议重构

目标：

- 在真实业务开发全面展开前，先把 `protocol` 收到更适合后续演进的形态
- 把 `Result` 改成显式 `ok/value/error` 结构
- 明确“简单对象 / 可桥接对象”的合同
- 把 `ObjectUtils`、`NumberUtils`、`StringUtils` 补到够用
- 为后续代码规范讨论建立新基线

完成标准：

- 新 `Result` 结构定稿
- `protocol` 和 `std` 的边界重新写清
- `ObjectUtils`、`NumberUtils`、`StringUtils` 的最小范围写清
- 代码规范讨论已经被放到这一步之后，而不是之前

本阶段的具体工作：

- 把 `Result` 改成显式 `ok / value / error` 结构：
  - `{ ok: true; value: T; error: null }`
  - `{ ok: false; value: null; error: E }`
- 明确 `E` 的约束：必须是简单对象
- 定义“简单对象 / 可桥接值”的合同，标准至少包括：
  - 可序列化
  - 可桥接
  - 字段稳定
  - 不含函数
  - 不含 class 实例
  - 不含嵌套宿主对象
- 按新分工重写 `protocol` 和 `std` 的边界：
  - `protocol` 负责补齐 JS 缺的“类型与值协议”
  - `std` 负责补齐 JS 缺的“操作与能力”
- 在 `protocol` 中允许少量直接服务协议的最小运行时：
  - `PlainValue`
  - `PlainObject`
  - `BridgeableValue`
  - `Result` / `Option` / `None`
  - 数值域类型，例如 `SmiInt`、`FiniteFloat64`
  - 与这些协议直接绑定的最小判定/构造，例如 `isPlainObject`、`isBridgeableValue`、`isResult`、`ok(...)`、`err(...)`
- 在 `std` 中补齐三组能力主体：
  - `ObjectUtils`
  - `NumberUtils`
  - `StringUtils`

这阶段明确不做：

- 不把 `protocol` 长成工具库
- 不在 `protocol` 里放成套 `ObjectUtils` / `NumberUtils` / `StringUtils`
- 不借协议重构顺手重做 `automation` / `repo-command`

参考：

- [protocol-refactor.md](protocol-refactor.md)

## 阶段 3：内部代码规范定型

目标：

- 在进入真实业务开发前，先明确内部写法标准
- 明确什么时候用 `Option`
- 明确什么时候用 `Result`
- 明确 hot path 的写法
- 明确编排层和边界层的写法
- 明确注释标准

完成标准：

- 内部代码规范文档定稿
- 协议重构后的写法已经能映射到这份规范
- 规范已经足够指导真实业务开发

本阶段要定清的内容：

- 什么时候用 `Option`
- 什么时候用 `Result`
- `Option` 和 `Result` 的边界怎么分
- hot path 该怎么写
- 边界层该怎么写
- 编排层该怎么写
- 注释该怎么写
- 各包内部的写法基线怎么定

最低共识要求：

- “没有值”是正常情况，用 `Option`
- “出了问题”才用 `Result`
- 如果一个结果既可能缺失，又可能失败，优先写成 `Result<Option<T>, E>`
- hot path 优先追求 shape 稳定和调用点稳定
- 边界层必须当场解析、当场归一化、当场转成 Clover 协议
- 编排层负责组织步骤，不重新定义底层协议
- 注释只写原因、边界、约束、风险，不翻译代码

本阶段落地顺序：

1. 先让协议重构方案定型
2. 再让内部代码规范定型
3. 然后再把这些规范落进 lint / tsconfig / 测试
4. 最后再进入真实业务开发

参考：

- [code-conventions.md](code-conventions.md)

## 阶段 4：进入真实业务开发

目标：

- 先用现有能力做真实业务
- 不主动扩大 Clover 的产品面

完成标准：

- 业务开发已经开始使用 Clover
- 主线包能支撑第一批真实需求
- 没有因为“先做平台”耽误业务推进

## 阶段 5：按缺口补能力

目标：

- 只补真实业务暴露出来的能力缺口
- 每次只补最小必要面

完成标准：

- 每个新增能力都能对应具体业务问题
- 新能力有测试
- 新能力的落点说得清

## 阶段 6：阶段性收敛

目标：

- 把前一轮新增能力重新整理
- 判断哪些该留，哪些该降级，哪些该移走

完成标准：

- 包边界更清楚
- 文档和实现一致
- 仓库没有继续长出无主能力

## 当前明确不优先做的事

- 不先做大规模通用化
- 不先做新的包装层
- 不先把仓库私有工作流做成产品线
- 不先扩大量没有真实使用的 API
- 不继续围绕 `automation` / `repo-command` 扩主线

## 一轮开发结束后的检查

每轮结束后，都用这 5 条复查：

1. 这次改动是否来自真实业务
2. 新能力是否补得过大
3. 是否把仓库私有问题错误地升成产品问题
4. 测试和文档是否同步
5. 下一轮是该继续补，还是该先收
