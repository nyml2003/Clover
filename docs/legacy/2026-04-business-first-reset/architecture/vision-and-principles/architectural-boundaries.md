# 架构边界

先看粗粒度结论：Clover 有五类职责。更细的系统图见 `system-map.md`。

## 1. 协议核心

最底层只承载最小共享协议：

- `None`
- `Option<T>`
- `CloverError`
- `Result<T, Code, Payload>`
- 品牌类型和少量断言/守卫

这层必须保持极小、稳定、低依赖。

## 2. 通用运行时层

这一层提供围绕 Clover 协议展开的通用函数：

- 字符串与数字解析
- 对象与 guard 工具
- 路径和 URL 处理
- `Result` 组合函数
- 异步迭代辅助能力

它可以依赖协议核心，但不应把宿主边界杂质带进来。

## 3. 边界适配层

边界适配层负责把宿主世界翻译成 Clover 协议。

当前主要包括：

- Zod 解析结果到 `Result`
- `null` / `undefined` 到 `None`
- argv 到业务输入
- 业务结果到 CLI 输出
- Web request 到业务输入
- Web 业务结果到 response 输出

这层可以接触第三方库和宿主 API，但不应把这些依赖反向渗透进核心层。

## 4. 约束治理层

这层不提供业务能力，负责把设计主张制度化：

- tsconfig 预设
- ESLint 自定义规则
- ESLint 组合配置

它的职责是把“推荐做法”收紧成“默认做法”。

## 5. 自动化与入口层

这一层负责两件事：

- 维护和编排工程任务
- 把底层能力装成可直接调用的入口

当前主要包括：

- 单元测试
- benchmark
- 构建脚本
- 生成物清理脚本
- 文档
- 仓库工作流入口

短期内，这一层仍主要由 `scripts/` 里的 Python 脚本承担。长期方向不是一直保留 Python，而是把自动化基座逐步收进 `@clover.js` 体系。

依赖方向应该始终保持单向：

- 协议核心不依赖上层
- 通用运行时建立在协议核心之上
- 边界适配建立在协议与运行时之上
- 治理层横切这些层做约束
- 自动化与入口层建立在底层能力之上，但不能反向污染协议核心

边界纪律的关键不是“目录怎么摆”，而是高层能力不能反向污染低层协议。
