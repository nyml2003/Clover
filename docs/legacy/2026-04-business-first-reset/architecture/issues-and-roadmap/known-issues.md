# 已知问题

## 1. 静态规则只覆盖了一部分设计主张

当前 ESLint 规则已经能约束很多关键坏味道，但仍有不少系统级约束尚未被工具化，例如：

- 数值域品牌的一致使用
- 更高层的 shape 稳定性约束
- 包装层是否把宿主协议泄漏进核心域

这意味着 Clover 仍然依赖“文档 + code review”共同维持设计纪律，工具链还没有覆盖全部关键主张。

## 2. 系统级验证还偏轻量

仓库现在已经有：

- 最小 CI
- `pnpm test:system`
- benchmark 基线约定

但系统级验证仍然偏轻量，主要问题是：

- 还没有覆盖更多真实消费场景
- 还没有形成更完整的跨包集成验证矩阵
- benchmark 仍以场景约定为主，尚未形成长期趋势记录

这意味着当前系统级可信度已经起步，但离更成熟的回归防线还有距离。

## 3. 发布自动化仍未建立

仓库现在已经有版本与 release 文档，也有 `pnpm release:check` 聚合检查入口，但还没有真正的发布自动化闭环，例如：

- 自动生成 release notes
- 自动版本推进
- 自动发布流程

这意味着 Clover 现在已经具备发布治理规则，但还没有发布治理自动化。
