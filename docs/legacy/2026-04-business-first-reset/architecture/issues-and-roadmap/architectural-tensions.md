# 架构张力

## 1. 统一协议 vs 现实便捷写法

设计上，Clover 想把空值、错误和数值域统一成明确协议。

现实中，部分 API 仍然保留宿主原生表达，例如 `null` 和普通 `number`。这让系统在“协议纯度”和“实现直接性”之间保留了张力。

## 2. 可发布产品 vs 工作区内部耦合

设计上，Clover 想成为一套可复用的 SDK 与约束体系。

现实中，一些关键能力还停留在 monorepo 内部耦合状态，例如 config 直接引用 plugin 源码、多个工作区包标记为 `private`。这说明系统更接近“内部统一开发”而不是“已完成产品化分层”。

## 3. 静态保证 vs 当前工具覆盖

设计文档的主张很强，尤其是在：

- 禁空值双轨制
- 禁异常控制流
- 固定 shape
- 受控数值域

现实中，工具链已经覆盖了一部分，但尚未形成对全部主张的自动化守门。因此设计强度和工具强度之间仍不对齐。

## 4. 性能导向 vs 标准库扩张

设计上，Clover 强调极薄运行时和 V8 友好路径。

现实中，标准库已经开始覆盖更多通用函数和异步工具。方向本身没有问题，但这要求后续更清楚地区分：

- 哪些是稳定核心协议附近的能力
- 哪些是偏宿主便利性的扩展函数

否则标准库会逐渐接近一个“价值观明确但边界变宽”的通用工具库。

## 5. 规范先行 vs 现状文档滞后

过去的文档主要擅长定义终态，而不擅长解释当前状态和偏差。

这会导致两个后果：

- 新读者容易看到强规范，却不知道哪些已经落地
- 维护者容易知道代码现状，却缺少一份能系统承接审查结论的知识库

这次新增的 `docs/architecture/`，本身就是在修复这条张力。
