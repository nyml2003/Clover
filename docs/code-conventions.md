# 内部代码规范

这份文档只回答内部开发时怎么写代码。

目标不是把风格说漂亮。
目标是让团队内部先有一致标准。
只有内部标准稳定了，外部使用时才有可解释的约束。

## 适用范围

这份规范优先约束：

- `protocol`
- `std`
- `zod`
- `cli`
- `http`
- 仓库内部工具链与编排代码

## 1. 什么时候用 `Option`

`Option<T>` 用在“这是正常状态分支，不是错误”的场景。

适用：

- 字段可能缺失，但缺失本身不是失败
- 查询结果可能不存在
- 某个派生值可能暂时没有
- 核心对象里需要固定 shape，但某个槽位当前为空

不适用：

- 参数校验失败
- 业务规则失败
- I/O 失败
- 任何需要携带失败原因的场景

一句话判断：

- “没有值”是正常情况，用 `Option`
- “出了问题”才用 `Result`

## 2. 什么时候用 `Result`

`Result<T, E>` 用在“调用可能失败，而且失败需要明确原因”的场景。

适用：

- 输入解析失败
- 规则校验失败
- 宿主调用失败
- 边界转译失败
- 编排步骤失败

要求：

- 失败分支必须有明确 `error`
- `error` 必须满足简单对象合同
- 同一个函数的失败语义要稳定，不要一会儿字符串一会儿对象

不适用：

- 只是字段为空
- 只是某个值缺席
- 只是还没算出来

## 3. `Option` 和 `Result` 的边界

不要混用。

默认规则：

- 缺失值：`Option`
- 失败值：`Result`

如果一个结果既可能缺失，又可能失败：

- 先用 `Result`
- 成功分支里再放 `Option`

也就是：

```ts
Result<Option<T>, E>
```

不要反过来写成“缺失值里再包失败”。

## 4. hot path 怎么写

hot path 的目标只有两个：

- shape 稳定
- 调用点稳定

默认要求：

- 输入 shape 先收紧，再进核心逻辑
- 尽量只处理单一值域
- 少分配临时对象
- 少临时闭包
- 少多态调用
- 少把字符串、数字、对象、空值混成一锅

优先做法：

- 小函数
- 显式参数
- 固定返回 shape
- 早归一化
- 早分层

避免做法：

- 一个函数同时吃很多种输入 shape
- 在热路径里频繁组装不同对象结构
- 把异常当普通分支
- 把 `undefined` / `null` / `Option` 混着用

## 5. 边界层怎么写

边界层负责接触宿主杂质。

包括：

- `unknown`
- `null`
- `undefined`
- Zod
- HTTP request / response
- argv
- 外部返回值

边界层规则：

1. 当场解析
2. 当场归一化
3. 当场转成 Clover 协议
4. 不把宿主语义带进核心层

边界层允许：

- `try/catch`
- `typeof`
- `instanceof`
- `zod`
- 宿主对象

但这些东西必须停在边界。

## 6. 编排层怎么写

编排层不是核心协议层。
它的职责是串步骤，不是发明新协议。

编排层规则：

- 负责组织步骤
- 负责组合已有能力
- 负责决定执行顺序
- 负责把失败向上汇总

编排层不该做的事：

- 重新定义底层协议
- 偷偷扩展边界语义
- 一边编排一边夹带底层工具逻辑
- 为了“方便”绕开 `protocol` / `std` 的约束

一句话：

编排层写流程，不写新地基。

## 7. 注释怎么写

注释不是翻译代码。
注释只写这几类信息：

- 为什么这样写
- 这里的边界条件是什么
- 这里的性能假设是什么
- 这里的协议约束是什么
- 哪些写法是刻意为之，不要随便改

不要写：

- 变量赋值解释
- 循环在循环
- 一眼能看懂的行为复述

好注释示例：

- 这里统一把宿主空值收敛成 `None`，后面的核心层不再接受 `null`
- 这里保留固定字段集合，避免热路径里切换 object shape
- 这里不用异常分支，统一走 `Result`，方便上层编排汇总失败

## 8. 包内写法基线

### `protocol`

- 写协议结构、值语义和最小判定/构造
- 可以有少量直接服务协议的运行时
- 不写宿主相关逻辑
- 不写成套 `ObjectUtils` / `NumberUtils` / `StringUtils`

### `std`

- 写围绕协议的通用工具实现
- 工具要薄
- 工具要能解释为什么值得进入 `std`
- `ObjectUtils` / `NumberUtils` / `StringUtils` 这类能力主体放这里

### `zod` / `cli` / `http`

- 专心做边界桥接
- 不反向定义核心协议
- 不长成第二套标准库

### 编排和内部脚本

- 以任务组织为主
- 不抢协议层和运行时层的职责
- 工具链脚本优先保持独立，不直接耦合当前项目包
- 可以重写一份同样风格的类型、判定和小工具
- 允许重复实现，只要这样能保持脚本体系独立
- 要复用的是规范和设计判断，不是当前仓库的实现绑定

### 独立脚本体系的额外要求

如果是替换当前 `scripts/` 的新脚本体系，还要额外满足：

- 不把 `@clover.js/protocol`、`@clover.js/std` 当成直接运行前提
- 不依赖当前 monorepo 的内部路径约定才能工作
- 可以借鉴当前协议和工具写法
- 可以复制必要类型和方法
- 但复制后的实现仍要遵守 Clover 的代码规范
- 可以作为新设想的试点位
- 新设想先在脚本里验证，再决定是否进入主线
- 不为了试点先反向修改主线包

这里“复制必要类型和方法”是允许的，不视为坏味道。

前提只有两个：

- 复制是为了保持脚本体系独立
- 复制后的实现继续遵守 Clover 的写法和约束

## 9. 什么时候要停下来复查

出现下面这些信号，要先停下来，不继续写：

- 开始分不清这是 `Option` 还是 `Result`
- 开始把缺失和失败混成一个概念
- 一个函数同时承担边界层和核心层职责
- 编排层开始偷写底层逻辑
- 为了方便开始牺牲 shape 稳定
- 注释开始变成翻译代码

## 10. 规范落地顺序

这份规范不是只放在文档里。

后续要按这个顺序落地：

1. 协议重构先定型
2. 这份规范定型
3. 再补 lint / tsconfig / 测试约束
4. 最后用真实业务继续检验
