# 实现形态

当前实现已经具备一个比较清晰的系统轮廓。

## 1. 运行时协议已经落地为极小核心

底层已经有稳定的最小协议面：

- `None` 单例哨兵
- `Option<T>`
- `CloverError`
- `Result<T, Code, Payload>`
- `createError`、`isError`、`isNone` 等基础能力

这说明 Clover 已经不只是概念，而是形成了可复用的协议底座。

## 2. 通用运行时层已经覆盖几个明确方向

当前标准库实现围绕以下主题展开：

- 字符串扫描与比较
- 数字解析和范围处理
- 对象 key/entry 辅助
- `Result` 组合工具
- 路径规整
- URL 解析与归一化
- 异步迭代辅助
- 基础类型守卫

这层的形态是“薄函数集合”，而不是容器对象或类层级。

## 3. 边界适配层已有可用最小闭环

当前边界能力已经形成基础闭环：

- Zod `safeParse` 结果可转为 Clover `Result`
- `optional` / `nullable` 输入可以收敛到 `None`
- CLI 层可以读取 argv、渲染错误、映射 exit code

这表明 Clover 不只是在定义协议，也开始承接真实宿主边界。

## 4. 工具治理层已开始制度化

当前已经有：

- 3 套 tsconfig 预设
- 一组 Clover 自定义 ESLint 规则
- 一套 flat config 组合配置

规则重点集中在：

- 禁危险语法
- 禁某些与 Clover 协议冲突的写法
- 约束 `createError(...)` 的 code 与 payload

## 5. 验证体系已具备基本可信度

当前仓库已经具备：

- `vitest` 单元测试
- 覆盖率门槛配置
- `lint`
- `build`
- `bench`

其中 benchmark 已经开始拿当前真实 API 和原生实现做对照，而不是空壳占位。

## 6. 文档状态仍然偏“规范先行”

现有文档更像设计规格和原则陈述，适合定义方向；对“现在做到哪一步”与“哪里有偏差”的说明此前不足，这也是新增本目录的直接原因。
