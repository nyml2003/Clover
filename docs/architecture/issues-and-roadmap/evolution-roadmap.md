# 演进路线

这份路线图按系统层次排，不按目录排。

先说结论：

- 先稳住现有核心能力栈
- 再把宿主逻辑边界拆清楚
- 然后长自动化基座
- 最后再把仓库工作流从 Python 迁进 `@clover.js`

这里说的 Web，不是渲染层。这里先通过 HTTP 请求响应边界来承接 Web 宿主里的逻辑层和边界层。

这里说的自动化基座，不是 `@clover.js/cli` 的一部分。它是一条单独的上层能力线。

短期判断也先写清楚：

- 当前仓库工作流继续使用 Python
- 不急着马上替换根级脚本
- 先把长期结构长对，再做入口切换

## 一条总原则

每一层只做自己的事。

- `protocol` 只做最小协议
- `std` 只做通用运行时
- `zod` 只做通用边界适配
- `cli` 只做终端宿主边界
- `http` 只做 HTTP 请求响应边界
- future 自动化基座只做工程任务编排

最需要防住的两件事：

- 不要把宿主逻辑塞回 `std` 或 `protocol`
- 不要把仓库维护能力直接塞进 `@clover.js/cli`

## 阶段 0：稳住当前基线

这一阶段先不扩新线。先把已经有的东西稳住。

目标：

- 让现有核心能力栈继续可维护
- 让新文档和当前实现不打架
- 给后面的 Web 线和自动化线留出清楚边界

主要工作：

- 继续维护 `@clover.js/protocol`、`@clover.js/std`、`@clover.js/zod`、`@clover.js/cli`
- 继续维护 `@clover.js/tsconfig`、`@clover.js/eslint-plugin`、`@clover.js/eslint-config`
- 持续修正文档和实现漂移
- 明确哪些包是当前稳定消费面，哪些还是仓库内配套能力

完成标准：

- 根级 `build`、`lint`、`test` 稳定
- 架构文档能准确描述当前系统结构
- 新增能力不会继续打破现有边界

## 阶段 1：收紧现有宿主边界

这一阶段不先做新产品线，先把已经有的宿主边界收紧。

目标：

- 让 `@clover.js/cli` 继续保持薄
- 把“终端边界”和“工程编排”明确拆开
- 为后面的 Web 线和自动化线定出统一边界模型

主要工作：

- 审核 `@clover.js/cli` 的职责，只保留：
  - argv 读取
  - 结果渲染
  - stderr / stdout
  - exit code
  - Zod 桥接
- 不把 workspace 发现、任务图、构建编排、子进程执行放进 `@clover.js/cli`
- 补清 CLI 边界层的输入输出约束和错误约束
- 补清通用边界适配层和宿主边界层的分工

完成标准：

- `@clover.js/cli` 的职责边界清楚
- 终端宿主边界和工程任务编排不再混谈
- 后续再长新包时，不需要回头重定义边界

## 阶段 2：扩 HTTP 请求响应边界

这一阶段继续把第二条宿主线长完整。

目标：

- 把已经起步的 HTTP 请求响应边界继续长完整
- 给 Web 宿主里的逻辑层一个更清楚的落点
- 继续坚持“不碰渲染层”

主要工作：

- 先覆盖最核心的 HTTP 逻辑边界能力：
  - request / response 输入输出收敛
  - query / headers / cookies / body 解析
  - session / auth context 注入
  - handler / middleware / action 的逻辑骨架
  - Web 错误到 Clover 协议的转换
- 不在这一阶段做：
  - JSX
  - 页面渲染
  - 样式系统
  - 路由框架封装

完成标准：

- HTTP 请求响应边界和渲染层边界清楚
- HTTP 线建立在现有协议、运行时和边界适配之上
- 不会因为做 Web 线把系统重心带向 UI 框架

## 阶段 3：长出自动化基座

这一阶段开始补工程任务这条线。

目标：

- 把 repo workflow 的长期落点先长出来
- 让自动化能力进入 `@clover.js` 体系
- 但短期不强切 Python

主要工作：

- 新增自动化基座包
- 第一批只做最核心能力：
  - 子进程执行
  - 命令结果建模
  - 任务定义
  - 任务依赖图
  - workspace 发现
  - 日志和退出码
- 基于自动化基座，逐步补齐：
  - build
  - lint
  - typecheck
  - test
  - bench
  - release-check
- 明确这层是仓库维护基座，不是 `cli` 本体

短期策略：

- Python 继续作为当前稳定入口
- TypeScript 版本先并行长能力
- 先让结构长对，再让入口切换

完成标准：

- 自动化能力不再只能写在 `scripts/` 里
- 自动化基座和 `@clover.js/cli` 的边界清楚
- TypeScript 版本已经能覆盖主要工作流，但还不要求立刻替换 Python

## 阶段 4：切仓库入口

这一阶段才开始动根级工作流入口。

目标：

- 给 Clover 仓库一个统一命令入口
- 逐步把根脚本从 Python 切到 `@clover.js`
- 保持迁移期间可回退

主要工作：

- 基于自动化基座提供统一的 `clover` 命令入口
- 先让根级脚本调用 `clover`
- 再让包级任务入口逐步切过去
- Python 暂时保留作过渡和对照
- 等 TypeScript 入口跑顺后，再决定哪些 Python 脚本可以删除

完成标准：

- `build`、`lint`、`typecheck`、`test`、`bench`、`release-check` 都能走统一入口
- 根级工作流不再强依赖 Python 脚本
- 迁移期间出现问题时仍有清楚回退路径

## 阶段 5：进入产品化收敛

这一阶段才开始谈更强的产品面。

目标：

- 明确哪些上层能力要继续孵化
- 明确哪些可以进入对外消费面
- 把系统从“仓库内好用”推进到“结构上可复用”

主要工作：

- 审视 HTTP 请求响应边界是否已经稳定
- 审视自动化基座是否仍只服务 Clover 仓库，还是值得继续抽象
- 明确：
  - 哪些包继续 `private`
  - 哪些包进入受控消费
  - 哪些包以后可以进入稳定消费面
- 为新增上层包补齐：
  - README
  - 使用方式
  - 测试
  - 版本策略

完成标准：

- 新的上层能力不再只有仓库内部人才看得懂
- 对外承诺面和仓库私有能力分得更清楚
- 文档、包元信息和真实定位一致

## 这条路线刻意不先做什么

短期不先做这些事：

- 大规模扩 `std` 的 API 面
- 把 Web 线做成渲染层框架
- 把仓库编排直接塞进 `@clover.js/cli`
- 为了迁移而马上删除 Python
- 在自动化基座还没长出来之前就先切根入口

原因很简单：

- 基础层还没收稳时，扩 API 只会放大边界混乱
- Web 线一旦直接碰渲染层，系统焦点就会偏
- 自动化基座没长出来前，硬切 Python 只会把压力转成维护成本

## 建议执行顺序

接下来的工作按这个顺序排：

1. 继续稳住现有核心能力栈
2. 收紧 `@clover.js/cli` 的职责边界
3. 补 HTTP 请求响应边界
4. 补自动化基座
5. 再切仓库统一入口
6. 最后再决定 Python 的退场方式

这就是当前最稳的推进顺序。

它的核心好处只有两个：

- 先把系统长对
- 再把入口换掉
