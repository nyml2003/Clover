# 演进路线

这份路线图不按“包”拆，而按 Clover 作为一个整体系统的成熟顺序来规划。

总体原则固定为：

1. 先修仓库可信度
2. 再修协议一致性
3. 然后做产品化装配
4. 最后扩大验证深度和发布准备度

当前不建议先追求：

- API 面快速扩张
- 大量新增标准库函数
- 为了对外展示而先做示例工程

这些都应该排在系统基础稳定之后。

## 阶段 0：建立稳定维护基线

这一阶段的目标是把 Clover 从“可运行仓库”推进到“可稳定维护仓库”。

### 目标

- 让 fresh checkout 的验证结果稳定可复现
- 让仓库具备最小 CI 信号
- 让维护顺序和失败定位更确定

### 主要工作

- 修复根级 `pnpm typecheck` 对已有构建产物的顺序依赖
- 增加最小 CI 流水线，至少覆盖：
  - 安装
  - `pnpm test`
  - `pnpm lint`
  - `pnpm build`
  - `pnpm typecheck`
- 把当前维护 SOP 和文档入口接入根文档，减少“只有仓库熟人知道顺序”的隐性知识
- 明确 clean checkout 下的推荐验证顺序
- 明确哪些命令是开发期入口，哪些命令是发布前入口

### 交付结果

- 根级命令在干净工作树上稳定通过
- 仓库存在明确的 CI 配置
- 文档中有统一的维护入口和验证顺序

### 完成标准

- 新 clone 的仓库不需要先手工构建局部包，根级验证就能跑通
- 任意维护者都能只看文档就知道正确验证顺序
- CI 失败时能直接定位到 lint、test、build 或 typecheck 的具体阶段

## 阶段 1：收敛协议与实现现实

这一阶段的目标是把实现继续拉回 Clover 自己定义的协议模型，减少“文档一套、代码一套”。

### 目标

- 统一空值协议
- 明确数值域边界
- 明确边界层与核心层的实际分工

### 主要工作

- 收敛 std 层残留的 `null` 暴露，优先处理 URL 相关 API
- 明确哪些场景允许宿主值形态停留在边界层，哪些场景必须收敛到 `Option<T>`
- 为数值品牌类型补齐更清晰的使用边界，避免 brand 只停留在类型声明层
- 审核当前 std / zod / cli 的 API 面，区分：
  - 核心协议表达
  - 边界适配表达
  - 只是实现方便的历史残留
- 把已确认的协议约束继续前移到 lint 或测试，而不是只留在文档

### 交付结果

- 新增 API 默认遵守 `None` / `Option<T>` 空值模型
- 历史 API 中与协议主张冲突的部分被记录、迁移或收敛
- 文档中的协议约束和代码中的实际公开类型更一致

### 完成标准

- 核心运行时公开 API 不再新增 `null` / `undefined` 语义漂移
- 已知协议例外要么被消除，要么被明确标记为边界例外
- 设计文档中最核心的协议主张，能在代码和工具层找到对应落点

## 阶段 2：把治理层做成真正可消费的产品

这一阶段的目标是把 Clover 从“仓库内部统一开发”推进到“外部项目可消费”。

### 目标

- 让治理层包具备稳定消费方式
- 明确 public/private 策略
- 减少工作区源码路径耦合

### 主要工作

- 去掉 `@clover/eslint-config` 对 plugin 源码路径的直接依赖
- 梳理所有 workspace 包的定位：
  - 对外发布
  - 内部孵化
  - 暂不承诺稳定性
- 为对外消费的包明确稳定入口、安装方式和组合方式
- 补齐 tsconfig / eslint-plugin / eslint-config 的消费文档
- 明确版本推进策略，至少区分：
  - 仓库内部可变接口
  - 对外承诺接口

### 交付结果

- 治理层包不再依赖 monorepo 内部源码布局才能工作
- 包边界和消费方式清晰
- 外部项目接入 Clover 的路径可描述、可复制

### 完成标准

- `eslint-config` 可以基于稳定插件入口工作，而不是引用 `src/index.ts`
- 对外包与内部包的状态在文档和 `package.json` 上都一致
- 一个新项目如何接入 Clover 的最小路径已经被文档化

## 阶段 3：建立系统级验证闭环

这一阶段的目标是让 Clover 不只“局部可用”，而是“系统性可信”。

### 目标

- 把验证从单元级推进到系统级
- 让 benchmark 真正承担回归参考作用
- 让文档、实现、验证形成闭环

### 主要工作

- 增加 clean checkout 验证场景
- 增加更高层的集成验证，重点覆盖：
  - 协议层到边界层的串联
  - tooling 包的实际装配
  - 构建产物与源码消费路径的一致性
- 为 benchmark 建立稳定输入集和对比基线
- 把性能敏感 API 的验证从“临时跑 bench”提升到“可重复比较”
- 建立文档回顾机制：每次重大实现变化都回看 architecture 文档

### 交付结果

- 验证不再只依赖单元测试通过
- benchmark 输出可比较、可追踪
- 架构文档开始持续反映仓库现实，而不是一次性审查产物

### 完成标准

- 至少存在一组系统级验证，能覆盖源码、构建产物和治理层装配
- benchmark 有稳定输入和可复用命令，不再只是临时观测工具
- 架构文档会随着系统行为变化同步维护

## 阶段 4：进入发布准备期

这一阶段的目标是为未来的对外稳定发布做最后收敛。

### 目标

- 明确 Clover 哪些部分准备进入稳定承诺
- 明确迁移策略和兼容边界
- 明确发布前检查项

### 主要工作

- 明确首批稳定包名单
- 为关键公开接口补齐迁移说明和兼容策略
- 明确 semver 规则，尤其是：
  - 类型变化算不算 breaking
  - lint 规则收紧算不算 breaking
  - 错误码和 payload 变化算不算 breaking
- 形成发布前检查清单：
  - 文档
  - 构建
  - 类型
  - 测试
  - benchmark
  - 包边界
- 如果决定对外发布，再补齐 release 流程和版本推进机制

### 交付结果

- Clover 有清晰的稳定面
- 破坏性变更的判断标准明确
- 仓库具备进入外部消费阶段的基础准备

### 完成标准

- 维护者对“什么是稳定承诺、什么仍在试验期”没有歧义
- 发布前检查项可执行，不依赖口头经验
- 文档、包元信息和实际装配方式一致

## 跨阶段约束

无论推进到哪个阶段，都坚持这些原则：

- 不为短期方便牺牲协议一致性
- 不在基础验证不稳定时扩 API 面
- 不用 `examples/` 驱动主系统设计
- 不让文档长期落后于实现
- 不把历史例外继续扩散成新常态

## 建议推进顺序

如果按实际执行拆成接下来的工作批次，建议这样排：

1. 修 `typecheck` 顺序依赖，并补最小 CI
2. 收敛 URL 等 std API 的 `null` 泄漏
3. 解耦 `eslint-config` 和 plugin 源码路径
4. 明确包对外策略并补消费文档
5. 增加系统级验证和 benchmark 基线
6. 最后再讨论稳定发布面和版本策略

这个顺序的核心原因很简单：

- 没有稳定验证，就没有可信演进
- 没有协议一致性，就没有稳定产品面
- 没有稳定产品面，就不该过早讨论发布
