# 已知问题

## 1. 对外消费文档和稳定承诺仍不完整

仓库当前已经能从 `package.json` 看出一条初步边界：

- `@clover/protocol`
- `@clover/std`

更接近对外消费面，而以下包仍属于仓库内配套能力：

- `@clover/zod`
- `@clover/cli`
- `@clover/eslint-plugin`
- `@clover/eslint-config`
- `@clover/tsconfig`

但当前仍缺少更清晰的外部说明，例如：

- 哪些包承诺稳定
- 哪些包仍处于孵化期
- 外部项目推荐如何接入
- 哪些变更会被视为 breaking

这意味着“当前状态”已经不算模糊，但“稳定承诺”仍然没有被正式文档化。

## 2. 静态规则只覆盖了一部分设计主张

当前 ESLint 规则已经能约束很多关键坏味道，但仍有不少系统级约束尚未被工具化，例如：

- 数值域品牌的一致使用
- 更高层的 shape 稳定性约束
- 包装层是否把宿主协议泄漏进核心域

这意味着 Clover 仍然依赖“文档 + code review”共同维持设计纪律，工具链还没有覆盖全部关键主张。

## 3. 发布和版本治理仍未建立

仓库现在已经具备最小 CI，但还没有真正的发布治理闭环，例如：

- 稳定包名单
- semver 约定
- breaking change 判断标准
- release 流程

这意味着 Clover 现在更适合持续内部收敛，而不是直接承诺外部稳定消费。
