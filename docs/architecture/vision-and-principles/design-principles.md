# 设计原则

## 1. 类型约束前移

能在类型层锁定的约束，不留到运行期猜。

这包括：

- 变量和值域
- 函数输入输出
- 对象 shape
- 空值协议
- 错误通道

## 2. 运行时表面极薄

允许存在少量必要运行时值，但不接受为表达类型语义再造一层运行时对象系统。

当前设计强调：

- `None` 作为单例空值哨兵
- 固定 shape 的 `CloverError`
- 极薄的构造和判定函数
- 边界层使用 Zod，但不把它带进核心域

## 3. 固定 shape 与稳定协议

Clover 的对象和错误都追求稳定 shape，而不是动态扩展。

这背后的意图是同时满足两件事：

- 让代码在语义上更可预测
- 让引擎更容易维持单态和稳定优化路径

## 4. 边界层与核心层分离

宿主输入天然混乱，核心域不应该直接暴露在这种混乱下。

因此 Clover 把职责拆成两段：

- 边界层接触 `unknown`、`null`、`undefined`、Zod 和宿主 API
- 核心层只处理已经归一化后的稳定协议

## 5. 工具链即规范

规范不是单靠文档宣传，而是要进入工具链。

当前路线是：

- TypeScript 负责类型面和编译期收紧
- ESLint 负责危险语法和协议违规
- 测试和 benchmark 负责验证行为与性能判断

## 6. 高版本运行时优先

Clover 选择面向现代 Node 和现代 ESM，而不是为旧环境兼容牺牲设计空间。

这意味着：

- 倾向直接保留现代语法
- 避免无意义的降级包装
- 更关注现代 V8 的成本模型，而不是历史包袱
