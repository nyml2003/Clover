# 协议重构计划

这一步发生在“真实业务开发正式展开”之前。

原因很简单：

- `protocol` 现在的 `Result` 形态会影响所有上层包
- 对“什么是可桥接、可序列化的简单对象”还没有统一定义
- `number` 和 `string` 的基础处理能力还不够稳，后面越用越难补
- 代码规范也要和新的协议形态一起收敛

## 这一步要改什么

### 1. 重构 `Result`

当前方向改成：

```ts
type Result<T, E> =
  | { ok: true; value: T; error: null }
  | { ok: false; value: null; error: E };
```

目标：

- 让成功分支和失败分支 shape 更直接
- 减少对“错误对象本身兼任失败分支”的依赖
- 让上层边界和工具函数更容易桥接

### 2. 统一错误对象约束

`E` 不再是任意值。

当前方向：

- `E` 必须满足“简单对象”
- 简单对象的标准不是“看起来简单”
- 标准是：
  - 可序列化
  - 可桥接
  - 字段稳定
  - 不含函数
  - 不含 class 实例
  - 不含嵌套宿主对象

这里 `protocol` 负责定义这个合同。

### 3. 补基础工具能力

这里按一个更清楚的原则分层：

- `protocol` 负责补齐 JS 缺的“类型与值协议”
- `std` 负责补齐 JS 缺的“操作与能力”

这意味着：

- `protocol` 不只是纯 type
- `protocol` 可以有少量运行时
- 但这些运行时只服务协议成立，不长成通用工具库

所以：

- `ObjectUtils`
- `NumberUtils`
- `StringUtils`

这三组成套能力仍然建议放在 `std`。

而 `protocol` 里可以放：

- `PlainValue`
- `PlainObject`
- `BridgeableValue`
- `Result` / `Option` / `None`
- 数值域类型，例如 `SmiInt`、`FiniteFloat64`
- 与这些协议直接绑定的最小判定/构造，例如：
  - `isPlainObject`
  - `isBridgeableValue`
  - `isResult`
  - `ok(...)`
  - `err(...)`
  - 数值域的最小判定或构造

边界规则是：

- `protocol` 负责定义“什么值是合法的”
- `std` 负责定义“怎么操作这些值”

### 4. 补三组核心工具

#### `ObjectUtils`

目标：

- 明确定义简单对象
- 提供判断、校验、收紧、桥接辅助

重点不是花哨 API。
重点是围绕 `protocol` 已定义的简单对象合同，提供真正可用的对象操作能力。

#### `NumberUtils`

目标：

- 处理 `number` 默认是 double 带来的问题
- 帮助整数、小整数、有限浮点数走稳定路径
- 尽量命中 V8 优化友好的值域和调用方式

重点：

- 小整数判断
- 有限数判断
- 浮点污染隔离
- 热路径数值收紧

同时约束：

- 数值域类型和最小判定在 `protocol`
- 更完整的数值工具在 `std`

#### `StringUtils`

目标：

- 把字符串基础处理补到够用
- 明确 UTF-8 / Unicode 相关操作的边界

重点：

- UTF-8 相关的编码/解码桥接
- 字符串扫描和切片时的稳定行为
- 不把“按 code unit 处理”和“按字符处理”混在一起

`StringUtils` 主体放在 `std`。
除非后面明确需要某种字符串协议类型，否则 `protocol` 不承担字符串工具职责。

## 这一步不做什么

- 不先扩很多通用工具函数
- 不为了 API 漂亮先做大而全包装
- 不先做和真实业务无关的字符串/数字花活
- 不先把 `automation` / `repo-command` 一起重构

## 包职责调整

这一步之后，包职责更清楚：

- `protocol`：协议类型、值语义、最小判定/构造
- `std`：对象、数字、字符串等通用能力实现
- `zod` / `cli` / `http`：基于新协议继续做边界桥接

## 代码规范讨论的时机

代码规范必须发生在“正式进入真实业务开发”之前。

更合理的顺序是：

1. 先把 `Result` 和简单对象合同定下来
2. 再把内部代码规范定下来
3. 最后再补 lint / tsconfig / 代码风格规则

也就是说：

- 代码规范可以晚于协议重构
- 但不能晚于真实业务开发启动
