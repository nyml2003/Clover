# 开发策略

## 总原则

先解决真实业务问题。
不要先解决想象中的通用问题。

## 默认工作流

1. 先写清当前业务需求
2. 先判断现有能力能不能直接支撑
3. 不能直接支撑时，只补这次必需的最小能力
4. 交付后再判断这次补的东西要不要沉淀成稳定能力
5. 每做完一轮，再补文档、测试和边界判断

## 新能力准入条件

一个能力要进入主线，至少满足这几条：

- 有明确业务场景，不是猜测
- 现有层级里确实缺这块能力
- 能说明它该落在哪一层
- 能给出对应测试
- 能说明引入它后，主线是否变宽

只要上面有一条答不清，就先不要做大。

## 默认落点

- 协议问题，落在 `protocol`
- 通用运行时问题，落在 `std`
- 通用边界转译问题，落在 `zod`、`cli`、`http`
- 约束治理问题，落在 `tsconfig`、`eslint-plugin`、`eslint-config`
- 仓库私有工作流问题，先留在仓库内部，不默认升格成主线能力

## 文档要求

每次非琐碎改动，至少要同步这三件事：

- 改了什么
- 为什么要改
- 这次改动是业务补洞，还是主线收敛

## 停下来的条件

一轮开发先停，不继续扩，满足任一条就要停下来复盘：

- 这次业务缺口已经补上
- 新增抽象已经开始长第二层包装
- 文档和测试还没跟上
- 包边界开始变得说不清
