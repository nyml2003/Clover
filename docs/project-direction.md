# 项目方向

## 当前判断

Clover 现在不再优先扩通用能力面。

当前优先级是：

1. 先把仓库工具链方向收敛到 `tsx + esbuild`，并完全消除 Python 脚本
2. 再先做一轮协议重构
3. 然后进入真实业务开发
4. 用真实需求检验现有设计
5. 只在确实缺能力时补最小支持
6. 每补一轮，再回头收边界和整理

## 现在要做的事

- 让 `protocol` 和 `std` 继续稳定
- 在真实业务全面展开前重构 `Result`、简单对象合同和基础 utils 分工
- 保持 `zod`、`cli`、`http` 这些边界层可用
- 保持 `tsconfig`、`eslint-plugin`、`eslint-config` 这套治理链路能约束代码
- 把仓库内部脚本路线收敛到 `tsx + esbuild`，并移除 Python 脚本
- 先把脚本体系做成与当前项目包不耦合的独立实现
- 允许脚本体系重写一份相同思想的类型和方法
- 把独立脚本体系作为新设想的试点场
- 用这些能力支撑真实业务开发

## 现在不做的事

- 不先把 Clover 做成更大的通用产品
- 不先扩很多没有真实使用场景的新 API
- 不先把仓库内部工作流包装成对外产品线
- 不为未来想象中的需求先做抽象

## 成功标准

这阶段完成，不看“包变多了没有”，看这几件事：

- 至少有真实业务路径跑在 Clover 现有能力上
- 新增能力都能对应到明确业务缺口
- 每次新增后，仓库没有继续失控扩张
- 文档、测试和边界判断能跟上实现

## 失败信号

出现下面这些情况，说明又偏了：

- 还没遇到真实业务问题，就先做一轮泛化抽象
- 为了“以后也许会用”先长新包或新层
- 仓库私有工具开始抢主线
- 入口层和包装层扩得比核心能力还快
- `protocol` 开始长成工具库，或者 `std` 开始反向定义协议
- 为了复用当前包实现，把脚本体系重新绑死到当前项目结构
