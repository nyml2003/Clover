# Clover 版本与兼容性策略

这份文档定义 Clover 当前的版本判断规则。

## 1. 总原则

- 先定义稳定面，再谈稳定发布
- 兼容性判断以公开消费面为准，不以仓库内部实现细节为准
- 类型、运行时行为、错误协议和工具规则都可能构成兼容性问题

## 2. 当前适用范围

当前版本策略优先约束：

- `@clover.js/protocol`
- `@clover.js/std`

其他包仍处于孵化或仓库配套阶段，不默认承诺长期稳定。

## 3. Breaking Change 判定

对于当前稳定消费面，以下变化默认视为 breaking：

- 删除或重命名公开导出
- 改变公开函数签名
- 改变公开返回值协议
- 改变 `Option` / `Result` / 错误对象的公开语义
- 改变错误码
- 改变错误 payload shape
- 让此前合法的 TypeScript 调用方式变成类型错误
- 改变子路径导出或入口解析方式

以下变化通常不算 breaking，但仍应写入变更说明：

- 兼容性不变的内部重构
- 明确的 bug 修复
- 文档澄清
- 不改变公开契约的性能优化

## 4. 0.x 阶段的版本推进

在 Clover 仍处于 `0.x` 阶段时，采用以下策略：

- breaking change：提升 minor
- 向后兼容的新能力：提升 patch 或 minor
- 纯 bugfix / 文档 / 内部维护：提升 patch

等进入 `1.0.0` 之后，再切换到标准 major/minor/patch 语义：

- breaking change：major
- 向后兼容的新能力：minor
- 向后兼容的修复：patch

## 5. 工具链变化的兼容性判断

对于治理层包，虽然当前不作为稳定消费面承诺，但仍建议按以下规则判断影响：

- 新增 lint 报错，默认视为规则收紧
- 改变配置默认行为，默认视为消费行为变化
- 改变 tsconfig 默认严格度，默认视为类型行为变化

如果未来这些包进入稳定消费面，上述变化应纳入正式 breaking change 判断。

## 6. 变更记录要求

只要触及稳定消费面，就应该记录：

- 受影响的包
- 受影响的公开接口
- 是否 breaking
- 迁移方式
- 是否需要同步文档或 benchmark
